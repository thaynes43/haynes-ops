# Tier-4 component playbooks

Per-component upgrade knowledge for the irreducible-manual set — the supporting
helmrelease/values edits a PR check can't make, the post-upgrade health checks,
and the **adversarially-verified rollback** (break-glass steps flagged). Consumed
by the [upgrade-shepherd](upgrade-shepherd.md) (mode 3 edits) and the
[upgrade-health-gate](upgrade-health-gate.md) (per-component verify). Generated
2026-06-30 from the per-component spec + rollback-verification pass; keep in sync
with the manifests it cites.


These are the per-component sections for the storage, database, networking, and platform components in the irreducible-manual set. Each is consumed by the upgrade-shepherd (Tier-4) and the scheduled health gate. Shared mechanics — the `haynes-ops-bot` identity, `.renovate/holds.json5` protocol, the read-only Omni Reader SA (VERIFY only; never `flux reconcile --with-source`), and the batch process — live in [`docs/renovate/tier4-bot-setup.md`](../../docs/renovate/tier4-bot-setup.md), [`docs/renovate/README.md`](../../docs/renovate/README.md), and [`.agents/runbooks/renovate-upgrade-batches.md`](./renovate-upgrade-batches.md). Every `kubectl`/`flux`/`helm` write below is tagged **(BREAK-GLASS)** — it needs a privileged kubeconfig the scheduled shepherd does not hold.

## rook-ceph

Rook operator + CephCluster + CSI drivers — distributed block/fs/object storage. `ceph-block` is the **default StorageClass**, so this is the widest stateful blast radius in the fleet; a bad push can sever storage for every stateful app at once.

- **Renovate match:** datasource `docker`; packageNames `ghcr.io/rook/rook-ceph`, `ghcr.io/rook/rook-ceph-cluster`, `ghcr.io/home-operations/charts-mirror/ceph-csi-drivers`. All three charts are OCIRepository-sourced, so Renovate tracks them as **`docker`, not `helm`** (PR titles `fix(container): update image ghcr.io/rook/rook-ceph(-cluster)…`, e.g. #1863/#1864). The two rook charts share the v1.x line and **MUST move together as a pair** (operator tag == cluster tag); `ceph-csi-drivers` is an independent line (home-ops mirror, 1.0.1). No rook group rule and no autoMerge entry exist today, so operator and cluster arrive as **TWO separate PRs the shepherd must merge together**. No rook hold currently. **CRITICAL SILENT DEP:** there is no `quay.io/ceph/ceph` ref and no `cephClusterSpec.cephVersion` pin anywhere, so Renovate does NOT see the Ceph daemon version — a `rook-ceph-cluster` major silently carries a new default Ceph image (this is how Ceph rode Squid 19.2.3 → Tentacle 20.2.1 on v1.19→v1.20). Treat any cluster-chart major as a potential Ceph-major upgrade.

- **Repo paths:**
  - `kubernetes/main/apps/storage/rook-ceph/rook-ceph/ks.yaml` (3 KS in ns rook-ceph: `rook-ceph` → `rook-ceph-csi-drivers` → `rook-ceph-cluster`, wired by `dependsOn`; all `prune:false`)
  - `app/helmrelease.yaml` + `app/ocirepository.yaml` (operator, tag v1.20.1)

- **Post-upgrade health checks:**

  | Signal | Method | Query | Healthy | Benign-warn | Regression |
  |---|---|---|---|---|---|
  | Ceph overall health | promql | `ceph_health_status{cluster="rook-ceph"}` | `0` (HEALTH_OK) | `1` (WARN) when `ceph_health_detail` shows ONLY RECENT_CRASH (self-recovered mon.i/Mgr crash); clears on `ceph crash archive-all`, the CephDaemonCrash re-page is not new | `2` (ERR), or WARN >30m with a non-crash detail (OSD_DOWN, PG_AVAILABILITY, MON_DOWN, POOL_FULL, upgrade-stuck) |
  | Ceph daemon version convergence | promql | `count(ceph_osd_metadata) by (ceph_version)` (also mon/mds) | single version == intended (live: `20.2.1 tentacle`) | two versions transiently while daemons roll one-at-a-time | a Ceph MAJOR you didn't sign off (20→21) = STOP/page; or daemons split after the roll window |
  | MON quorum | promql | `count(ceph_mon_quorum_status == 1)` | `3` (b, i, j) | drops to 2 for <1-2m as a mon restarts in place | `<2`, or a mon flapping in/out across the window |
  | OSDs up/in + PG health | promql | `count(ceph_osd_up==1)`; `count(ceph_osd_in==1)`; `sum(ceph_pg_degraded)`; `sum(ceph_pg_down)` | 6 up, 6 in, degraded→0, down==0, active+clean | short degraded spike with backfill/recovery as an OSD restarts | OSD down >10m, `pg_down>0`, PGs stuck inactive, degraded not trending to 0 |
  | Flux reconcile state | flux | `flux get ks -n rook-ceph rook-ceph rook-ceph-csi-drivers rook-ceph-cluster; flux get hr -n rook-ceph` (read-only SA — no `--with-source`) | all 3 KS + 3 HR Ready=True on the new revision, in dependsOn order | cluster HR Reconciling/Upgrading while daemons roll (15m timeout) | any HR Ready=False `retries exhausted`, or csi-drivers HR not Ready (provisioner RBAC missing) |

- **Verify:**
  1. Confirm BOTH rook PRs (operator + cluster) moved to the **same** target tag; read release notes + `holds.json5` first. Merge together; let Flux pick up via poll/Receiver (read-only SA must NOT `flux reconcile --with-source`).
  2. Watch the order: `flux get ks -n rook-ceph -w` — operator Ready → csi-drivers Ready → cluster Ready.
  3. Watch daemons converge: `scripts/watchrook.sh` (or `kubectl -n rook-ceph get deploy -l rook_cluster=rook-ceph`) — every rook-version label flips, req/upd/avl equalize.
  4. Gate signals: `ceph_health_status==0` (or WARN=RECENT_CRASH only), quorum sums to 3, OSDs 6 up/in, `ceph_pg_degraded`→0, `ceph_version` uniform. (Optional **(BREAK-GLASS)** `kubectl -n rook-ceph exec deploy/rook-ceph-tools -- ceph -s && ceph versions` — prefer the PromQL signals.)
  5. CSI sanity: `kubectl -n rook-ceph get pods | grep -E 'csi-rbdplugin-provisioner|csi-cephfsplugin-provisioner'` all Running; grep their logs for `…-ctrlplugin-sa.* not found` (must be absent).
  6. Provisioning probe **(BREAK-GLASS — supervised session only)**: create one tiny PVC on `ceph-block` and one on `ceph-filesystem`, confirm Bound + a pod mounts each, then DELETE both in the SAME step (a leftover Completed pod pins `pvc-protection` → PVC stuck Terminating).
  7. Existing-mount sanity: spot-check a couple of stateful apps on `ceph-block` still read/write.
  8. Archive any RECENT_CRASH once stable **(BREAK-GLASS)** `ceph crash archive-all` → clears HEALTH_WARN so the alerts resolve instead of re-paging.

- **Rollback:** **Verdict: conditional convergence; data-loss risk LOW from the revert itself** (Rook *refuses* to downgrade Ceph daemons, so reverting the chart leaves the data plane physically untouched — it simply won't roll back). The real exposure is operator-/human-driven: (1) the common 2am case (Helm upgrade applied cleanly, Ceph regressed at **runtime**) gets ZERO auto-remediation — these HRs have no healthChecks/test/wait, so `strategy:rollback` never fires — and nothing auto-detects a *runtime* regression (the gate or a human must catch it), so storage stays degraded until a human pushes a revert (which is then pulled within the 30-min GitRepository poll — Flux here is poll-only, no active GitHub webhook); (2) HIGH/irreversible only via panic break-glass (wiping/recreating OSDs to "force" a Ceph downgrade can destroy a replica set); (3) deleting the default `ceph-block` SC to clear an immutability wedge blocks NEW PVC binds while absent. Net: **the git revert is safe; the danger is break-glass actions against the data plane.**

  Corrected procedure:
  1. **Triage with reads only; classify revertibility on "did the Ceph daemon IMAGE move AT ALL"** — not just majors (point-release daemon downgrades are also unsupported once `min_mon_release`/`require-osd-release` advance). Capture `flux get hr -n rook-ceph` (deployed chart versions), each HR's `status.history`, and `ceph versions` via the toolbox. Separately classify Helm-failure (HR Ready=False) vs RUNTIME (Ready=True but Ceph degraded) — if runtime, there is no auto-rollback net.
  2. **If the Ceph daemon image moved, OR the revert crosses the CSI architecture boundary, OR a CRD version changed — STOP.** One-way / structural. Page the human, FORWARD-fix (roll to a fixed Rook/Ceph release), and add a hold to `.renovate/holds.json5`.
  3. **Chart-logic-only regression:** `git revert <bump-sha>` (never bulk-checkout — drops mgr 4Gi, `ms_mode`, `requireMsgr2`). Keep operator+cluster on the same tag; revert csi-drivers only if it moved in the same batch. Commit + push.
  4. **Accept the ~30-min convergence floor.** Flux is poll-only (30-min GitRepository interval; no active GitHub webhook, internal-only receiver ingress) and the read-only SA can't force a source reconcile → up to 30m before the revert is even pulled, during which the default SC stays degraded. If unacceptable, escalate to **(BREAK-GLASS)** `flux reconcile source git haynes-ops`.
  5. **Verify the DEPLOYED chart, not just Ready.** Flux `lastAppliedRevision` tracks the git revision independently of the deployed Helm chart; `strategy:rollback` can report Ready=True while pinned on the bad release. Confirm via HR `status.history` / `helm history -n rook-ceph <release>` AND `ceph versions` (watch for refuse-to-downgrade spec/status drift), then the health signals + CSI probe.
  6. **(BREAK-GLASS) ladder** — by symptom: (a) Helm looping on an immutable object — `kubectl delete storageclass <ceph-block|-filesystem|-bucket>` so Helm recreates (ceph-block is default — recreate promptly), or `kubectl delete svc/deploy <name>` for an immutable Service clusterIP / operator selector; (b) csi-drivers stuck Ready=False with drivers torn down by `cleanupOnFail` — `flux suspend hr ceph-csi-drivers`, confirm git reverted, `flux resume`; (c) stuck Helm pending / rollback↔git fight — `flux suspend hr <name>` then resume, else delete the pending `sh.helm.release` secret; (d) CRD decode failures after a version-crossing revert — re-apply the target Rook CRDs before the cluster HR. **NEVER** `kubectl patch` the Flux-managed HR/CephCluster for a lasting change, and **NEVER** wipe/recreate OSDs or delete daemons to force a Ceph downgrade.
  7. **Shepherd policy:** on any cluster-chart major (or minor that could carry a Ceph point bump), detect the unpinned-`cephVersion` rider, confirm a single sequential step, and PAGE for the irreversible decision; record the outcome/hold.

  **Caveats:** Ceph majors (and same-major daemon downgrades) are **one-way** — merging a cluster major == accepting no Ceph downgrade. All 3 KS are `prune:false` (a revert won't prune rook resources — good — but leaves chart-removed resources orphaned on a downgrade; clean by hand). Do NOT rename/move the rook KS while rolling back (`flux-pvc-prune-safety`). Revert operator+cluster+csi-drivers as a consistent set. The operator auto-rollback reverts only the in-cluster Helm release, not Git.

## cnpg

CloudNativePG operator + the `postgres16` (3-instance HA) and `postgres16-pgvecto` (1-instance) Cluster CRs in ns `database`. Engine behind **authentik (SSO), grafana, immich, paperless-ngx, todos-for-dues** — an operator roll resets every consumer's connection via an unsupervised primary switchover.

- **Renovate match:** datasources `helm` + `docker`; packageNames `cloudnative-pg`, `ghcr.io/cloudnative-pg/postgresql`, `ghcr.io/tensorchord/cloudnative-vectorchord`. **Primary signal = the Helm CHART bump** (datasource `helm`, packageName `cloudnative-pg`, pinned at `spec.chart.spec.version` in `app/helmrelease.yaml`, currently 0.28.3; PR #1910 → 0.29.0; titles `feat|fix(helm): update chart cloudnative-pg…`). **The chart minor tracks the OPERATOR minor 1:1** — chart 0.28.3 = operator 1.29.1; chart 0.29.0 = operator 1.30.0 — so a "Tier-2-looking" chart minor is really an OPERATOR minor (read its release notes). **DATA-PLANE images are NOT auto-tracked:** `postgresql:16.4-34` (`cluster16.yaml`) and `cloudnative-vectorchord:16.13-1.1.1@sha256` (`cluster16-pgvecto.yaml`) have no `# renovate:` comment → zero PRs, bump by hand. Don't confuse with sibling postgres-shaped apps (dragonfly/emqx/immich's bundled pg). No cnpg on edge.

- **Repo paths:** `kubernetes/main/apps/database/cloudnative-pg/` → `app/helmrelease.yaml`, `app/helmrepository.yaml`, `app/externalsecret.yaml`, `app/prometheusrule.yaml`; `cluster/cluster16.yaml`, `cluster/cluster16-pgvecto.yaml`, `cluster/scheduledbackup.yaml`, `cluster/service.yaml`; `ks.yaml`.

  | Signal | Method | Query | Healthy | Benign-warn | Regression |

- **Verify:**
  1. `flux get hr cloudnative-pg -n database` → Ready=True, applied revision == merged chart.
  2. `kubectl get deploy -n database -l app.kubernetes.io/name=cloudnative-pg` → 1/1, image == new appVersion; `kubectl logs … --since=15m | grep -iE 'error|panic'`.

  4. `kubectl get cluster.postgresql.cnpg.io -n database` → both "healthy state", readyInstances == instances.
  5. PromQL: `cnpg_collector_up` all 1, `streaming_replicas{job="database/postgres16"}` back to 2.
  6. CNPG PrometheusRule alerts NOT firing (InstanceDown / ReplicaNotStreaming / ReplicationLagHigh / RestartLoop).
  7. Consumer smoke: Grafana queries, Authentik login, Immich/Paperless/todos healthy (connect via `postgres-lb 192.168.40.204` / `postgres.haynesops:5432`).
  8. Backups run: `kubectl get scheduledbackup,backup -n database` → a recent Backup Completed; no ReplicaNotStreaming (WAL archiving healthy).

- **Rollback:** **Verdict: conditional convergence.** Operator-chart revert alone is **low data-loss for the operator layer** (data plane PVCs/PGDATA untouched; instances are plain pods+PVCs, no StatefulSet selector trap). **HIGH/irreversible in three under-gated cases:** (1) the upgrade crossed a PostgreSQL **major** (`pg_upgrade` is one-way — reverting `imageName` to 16.x against migrated 17 PGDATA just crash-loops; restore from backup, don't revert); (2) a backup migration (in-tree barman → plugin) — reverting the operator without the cluster CR forks/orphans the S3 lineage and backups silently stop; (3) the reseed steps on **local openebs-hostpath** (single copy, node-pinned) — `kubectl delete pvc` for **pgvecto** = total data loss (single-instance, no streaming source).

  Corrected procedure:
  1. **Classify the regression first:** (a) control-plane only (operator crash-loop/RBAC/webhook, no instance roll yet) → git revert converges; (b) operator already rolled the instances and the new instance-manager misbehaves → git revert is necessary but likely needs break-glass; (c) data-plane (PG image/major, schema) → restore-from-backup, do NOT revert across a PG major; (d) backup migration → revert operator AND cluster together. Pick the path before touching git.
  2. **Git-first for (a)/(b)/(d):** re-pin `spec.chart.spec.version` to the prior chart (0.28.3 → operator 1.29.1), commit+push. For (b)/(d) also revert `cluster/*.yaml` in the SAME commit.
  3. **Set latency expectations:** Flux is poll-only (GitRepository 30-min interval, no active GitHub webhook), so the revert isn't pulled for up to 30m, then the cnpg Kustomization (15m interval) applies it and HelmController may re-roll 3 instances past the KS `wait:true, timeout:2m`. **Budget ~30-50m for full convergence; treat a spurious operator-KS "timeout" as expected. Do NOT escalate to break-glass before a full source-poll + reconcile + rollout has elapsed.**
  4. **If the bad upgrade FAILED:** let `strategy:rollback` auto-roll to 0.28.3 — but confirm the rollback's CRD re-apply (`crds.create:true` renders CRDs as templates) succeeded; if the HR goes Stalled on a CRD/SSA conflict, the git revert re-triggers it; only then reach **(BREAK-GLASS)** `kubectl apply --server-side` of the prior CRD.
  5. **Verify at steady state, not instantaneously:** wait until all `cnpg_collector_up==1` and `streaming_replicas==2` are sustained past the alert `for:15m` (the expected-replica denominator itself moves during the roll). Cross-check `kubectl get cluster -n database` directly rather than relying solely on Grafana — grafana's own DB connection blips during the switchover.
  6. **(BREAK-GLASS)** wedged Cluster after an operator-downgrade refuses the newer instance-manager (the EXPECTED path for case b): `kubectl cnpg promote <cluster> <instance>` or delete the stuck instance **POD** (not the PVC) to force re-attach.
  7. **(BREAK-GLASS)** reseed a broken **pg16 REPLICA only**: delete the replica POD first (streams from primary, no S3); only if the PVC is corrupt delete PVC+pod to force `pg_basebackup` from S3 (watch ~600GB egress). **GUARD: never apply this to pgvecto** (single-instance, local storage = data loss).
  8. **NEVER:** `kubectl delete cluster` to "reset" (cascades to PVCs on local storage); delete the pgvecto PVC; change `serverName`/`bootstrap.recovery.source` during a rollback (forks the lineage); reseed by recreating the whole Cluster (`bootstrap.recovery` runs only on empty PGDATA and would recover from v4 while re-archiving as v6 — timeline fork).
  9. **Pre-check deps:** confirm openebs and external-secrets/1Password healthy so `cloudnative-pg-secret` (superuser + AWS creds) is present — both DB auth and S3 restore gate on it.

  **Caveats:** operator downgrade is not upstream-supported (best-effort). PG data plane can't be downgraded across a major. CRD downgrade risk for a minor is the *opposite* of the usual fear — the older CRD's schema **drops** fields the newer operator added; grep the live Cluster spec for new-in-target fields before reverting (today's specs use only long-stable fields). Blast radius includes authentik (SSO) and grafana (the agent's own metric-read path) — a rollback that relies on reading grafana can blind itself during the switchover window.

Branch → PR → merge.
  4. **Let Flux apply (read-only):** GitHub Receiver / 30m poll. Do NOT `flux reconcile --with-source`. Watch `flux get ks dragonfly -n database` reaches Ready at the revert revision and the operator runs the prior image. If the HR is stuck "retries exhausted" and the reverted values equal the last auto-rolled-back state so Flux sees no drift, that is the one place you need **(BREAK-GLASS)** `flux suspend/resume hr dragonfly-operator -n database` to re-trigger (same for an INSTALL failure — `install.remediation` has no rollback).
  5. **Restore redis connectivity WITHOUT a write — the proven fix:** if immich/paperless still can't reach redis because the operator-owned `dragonfly` Service selector is wrong (the c2b3a743 failure mode) and the operator is wedged, **re-add a static `cluster/service.yaml`** (name `dragonfly`, port 6379, selector `app:dragonfly` + `app.kubernetes.io/name:dragonfly`, ClusterIP) and add it to `cluster/kustomization.yaml` — exactly as commit c2b3a743 did (remove again once the operator is back on a good version, per 3fea1ef0). This is a GIT change Flux applies under the read-only SA, independent of the operator reconciling. Safe because replicas:1.
  6. **(BREAK-GLASS)** only if STS immutable fields block convergence (older operator can't update a STS the newer one created with a different selector/serviceName/VCT): `kubectl delete statefulset dragonfly -n database` (cascade) and let the reverted operator recreate it — accept the keyspace flush. A `rollout restart` is NOT enough for an immutable-field conflict.
  7. **(BREAK-GLASS)** only for a genuine CRD apiVersion/structural incompatibility (NOT applicable today — single-version `v1alpha1`, no webhook; an SSA revert of the URL patches in place): `kubectl replace -f manifests/crd.yaml@<old-tag>` and verify stored CRs validate; otherwise leave the CRD forward and revert only the operator image.
  8. **Verify (read-only):** operator pod Ready on prior image; `dragonfly` Service has endpoints; immich-server + immich-ml ks Ready (un-gated); immich web + paperless reachable; expect a cold-cache/queue-rebuild blip if the data plane was flushed. Confirm operator↔server compat held in both directions for an operator-only revert.

  **Caveats:** in-memory = any restart flushes the keyspace. The SERVER image is a separate component (reverting the operator doesn't change it). CRD downgrade is one-way-ish for a future apiVersion bump. `app/rbac.yaml` is hand-maintained — keep it a forward superset on a revert (do not narrow). Flux won't force-replace an Established CRD via SSA.

A 2+-minor regression has no tested downgrade — escalate, don't improvise.

  5. **CATASTROPHIC CNI (severed) (BREAK-GLASS), ordered for what actually works:** the apiserver is reachable via KubePrism (`127.0.0.1:7445`, host-network) and cilium agents/operator use host-network, so (a) `kubectl -n kube-system set image ds/cilium cilium-agent=quay.io/cilium/charts/cilium:1.19.5 …` AND the operator Deployment image to the good tag — **this is the real downgrade, NOT a `rollout restart`** (a restart re-pulls the SAME bad image); or `helm rollback cilium <good-rev> -n kube-system`. The ONLY "loss" is transient DNS **availability** during pod churn or a Service-rename window. The danger is a **deadlock**: a bad push that breaks cluster DNS can stop source-controller from resolving `ghcr.io` to **re-pull the reverted chart** (the old tag's artifact was overwritten when the tag moved), so the git path can't apply.

  Corrected procedure:
  1. **Triage first — branch on whether CoreDNS is still resolving.** Use Grafana/Prometheus `coredns_build_info` + SERVFAIL rate, or any pod nslookup. "Resolves-wrong-but-up" → git path works; "fully down / CrashLoop" → in-cluster rollback first.
  2. **CASE A — UP but resolving wrong** (Corefile/value semantic bug, pods healthy, no auto-rollback): **git-first is canonical and DOES converge.** Revert the tag bump (`git revert <bump-sha>` or edit `ref.tag` → last-good, e.g. 1.45.2) in `kubernetes/main/.../coredns/app/ocirepository.yaml`; **MIRROR the same edit to `kubernetes/edge/.../coredns/app/ocirepository.yaml`** (separate file — a single revert may miss it; the bump+edge-sync landed in separate commits ed65b291/5a43cd97). Open PR; the **flux-local gate MUST confirm** the rendered Service still has `name=kube-dns` + `clusterIP=10.43.0.10` and an intended-only Corefile diff. Merge. The GitHub Receiver fast-tracks the push; OCIRepository re-pulls on its 15m interval, helm-controller upgrades on its 30m — because DNS still resolves, the read-only SA path converges with no write (worst case ~15-30m). Don't rely on auto-rollback here (a Helm-healthy-but-wrong upgrade doesn't trip it).
  3. **CASE B — FULLY DOWN / CrashLoop** (bad binary, unparseable Corefile, removed plugin): the git path can DEADLOCK (source-controller can't resolve ghcr.io to re-pull). Recovery comes from **in-cluster history, not git**: (1) PREFER letting Helm `strategy:rollback` complete — it uses the in-cluster release secret `sh.helm.release.v1.coredns.vN` (no DNS/pull) and should restore the prior good pods; verify it fired. (2) If it didn't, **(BREAK-GLASS)** `kubectl rollout undo deploy/coredns -n kube-system` (prior ReplicaSet, image already cached in containerd on the control-plane nodes) OR `helm rollback coredns <prev-rev> -n kube-system`. **Do NOT** `kubectl rollout restart` — it recreates the SAME broken pods with the bad image/Corefile.
  4. **CASE B cont.:** once resolution is restored by the in-cluster rollback, THEN do the Case-A git revert so declared state matches and the flap stops (auto-rollback parks on the good release but Git still declares the bad tag; any input change re-triggers the failing upgrade).
  5. **Wedged Helm release (any case):** if stuck `pending-upgrade` ("another operation in progress"), suspend/resume alone will NOT clear it. **(BREAK-GLASS)** `flux suspend hr coredns -n kube-system`, then `helm rollback coredns <last-deployed-rev>` (or delete the `sh.helm.release.v1.coredns.vN` pending-upgrade secret), then `flux resume hr coredns` after the git revert has landed.
  6. **Verify recovery:** `coredns_build_info` back to last-good; reload/panic counters flat at 0; SERVFAIL at baseline; functional nslookup of an in-cluster AND an external name; both main and edge `ocirepository.yaml` read the last-good tag; Flux OCIRepository + HR Ready=True on the reverted revision.

  **Caveats:** node/host DNS and the kube-apiserver are unaffected by CoreDNS (it serves POD DNS only), so kubectl/talosctl/Omni access survives the outage — recovery is always possible. **Never** change `service.name`/`service.clusterIP` (10.43.0.10 is immutable, baked into every resolv.conf). **Never** delete the coredns directory as a "rollback" — `ks.yaml prune:false` won't remove the running Deployment AND leaves it unmanaged; always a tag/value change. **Never** set HelmRelease `wait:false` to quiet a flap — `wait` defaults to true and is WHY auto-rollback triggers; disabling it silences the readiness gate and the safety net. Immutable `Service.spec.clusterIP` + `Deployment.spec.selector`: a forward change that touches either FAILS at apply and trips auto-rollback (safe); the real hazard is a Service-RENAME (kube-dns→coredns) opening a cluster-wide gap — the flux-local rendered-Service check is the pre-merge gate that must catch it.

2. **PATCH path (agent can fully own):** git revert the one-line `chart.spec.version` bump in `app/helmrelease.yaml` (or re-pin last-good), commit, push. Then WAIT and OBSERVE only — do NOT `flux reconcile --with-source` or any kubectl write **(both BREAK-GLASS)**. The revert resets Flux's retry counter and the HR re-upgrades-down on its ≤30m interval (old chart already cached). Expected unattended convergence ≤~30m.
3. **VERIFY** via the EXISTING Gatus external check + a real login + one proxied-app fetch — NOT pod-ready. Pod-ready false-greens on: redis-master wedged, blank-CSP login page, and outposts still on the new image.
4. **BEFORE declaring "stuck",** read HR conditions for the two Helm-layer wedges the naive procedure misses: (a) `field is immutable` on StatefulSet `authentik-redis-master` (Bitnami subchart selector/PVC) and (b) `another operation in progress` / pending-upgrade. Either → **(BREAK-GLASS)** human: `kubectl delete statefulset authentik-redis-master --cascade=orphan` then let Helm recreate, or `flux suspend/resume hr authentik` / delete the pending Helm release Secret. Agent stops and escalates.
5. **CONFIRM outposts followed core back:** they are DB-driven by the server (`container_image_base`), NOT git-revertable; they only return to the old image AFTER the server is healthy. A lagging outpost 502s every proxied IngressRoute even though the chart was reverted.
6. **MAJOR path:** do NOT expect git-revert alone to converge — DEFAULT TO FORWARD-FIX (re-run the sequential intermediate-major path). Reserve rollback for when forward-fix is impossible.
7. **MAJOR rollback (human, BREAK-GLASS, never the agent):** CNPG PITR-CLONE to a NEW Cluster (new manifest, INCREMENTED `serverName`, recoveryTarget = pre-upgrade) — NEVER edit `cluster16.yaml` `bootstrap.recovery` IN PLACE (`primaryUpdateStrategy:unsupervised` + `reusePVC` endangers grafana et al.). Then `pg_dump` authentik from the clone → drop/recreate the authentik DB in the SHARED cluster → `pg_restore` (don't strand authentik on the orphan clone via `AUTHENTIK_POSTGRESQL__HOST`). Accept loss of all authentik writes since the upgrade. Roll managed outposts back only after core is healthy.

Caveats: forward-only migrations (no down-migrations); shared Postgres + NO app-local VolSync (rollback is CNPG PITR into a SEPARATE cluster, not in-place, and loses identity/session changes since the upgrade); no Flux auto-rollback (no `strategy:rollback`); SSO blast radius — do rollbacks/upgrades in a low-traffic window. authentik ships NO CRDs (no stored-version/conversion trap) and its core Deployments have stable selectors (clean image revert); the only real ordering concern is the target's min-PostgreSQL on a FORWARD-fix (we run PG16), never on rollback. Read-only Reader SA can do the git revert + wait on the Flux poll only — every reconcile/scale/rollout-undo/restore is BREAK-GLASS.

**Rollback** — *Adversarial verdict:* **converges = conditional; data-loss = none for persistent state** (stateless DaemonSets/operator, no PVC/DB/schema/one-way migration). Two states git-revert CANNOT recover: a GPU physically off the PCI bus (host reboot, out-of-band) and a `generic-device-plugin` `:latest` regression (git never recorded the good digest). Availability urgency: intel = HIGH (Plex down); nvidia/generic = lower.

Corrected procedure:
0. **CLASSIFY the regression FIRST:** (i) Helm green but devices not registered → needs git revert (no auto-rollback fired); (ii) Helm upgrade FAILED → `strategy:rollback` already reverted the in-cluster release to last-good — confirm before touching git; (iii) HR stalled "retries exhausted" / "another operation in progress" → WEDGED, git revert alone won't converge → break-glass; (iv) capacity gone but HR/pods healthy → suspect GPU off the PCI bus (host reboot, out-of-band).
1. **GIT-FIRST, revert the WHOLE commit.** Prefer `git revert <sha>` over a surgical `chart.spec.version` re-pin — Renovate bumps frequently pair the version with `values:`/CR/NodeFeatureRule edits (nvidia flags→config-file) that a version-only re-pin strands against the old chart. Known-good pins: nvidia 0.19.3, intel operator+gpu 0.36.0 (BOTH, same version), NFD 0.18.3. generic-device-plugin has NO pinned version → cannot be git-reverted (step 6).
2. **BEFORE asserting a clean revert,** check whether the bump crossed a CRD apiVersion boundary (NFD `NodeFeatureRule`; intel `GpuDevicePlugin`/`DeviceConfig` — both charts use `crds:CreateReplace` on upgrade, so the revert REPLACES the CRD back to the OLDER schema). Diff the chart's `crds/` across versions: if a served/stored apiVersion changed, a clean downgrade is BLOCKED by `status.storedVersions` → **(BREAK-GLASS)** manual `kubectl patch` of storedVersions. Minor-only bumps (0.18.x/0.36.x) are safe.
3. If a known-bad release, add a hold to `.renovate/holds.json5` (EMQX-entry convention: `allowedVersions` excluding the bad version + Reason/Issue/Resume), validate `npx --yes --package renovate renovate-config-validator .renovate/holds.json5`, commit.
4. **Commit + push.** Do NOT `flux reconcile --with-source` **(BREAK-GLASS)**. The main github-receiver pokes the GitRepository on push; leaf Kustomizations + helm-controller reconcile on source/generation change — minutes (fallback 30m poll if the webhook/Traefik is degraded).
5. **For an INTEL revert specifically** (dependsOn forces operator-up-before-gpu; gpu DaemonSet is OPERATOR-managed): after the revert lands, verify the operator reconciled the existing `GpuDevicePlugin` CR and the DaemonSet image actually rolled back. A stale operator-left DaemonSet → **(BREAK-GLASS)** delete the DaemonSet / re-annotate the CR for Helm ownership.
6. **generic-device-plugin** is `:latest` and UNVERSIONED — nothing to git-revert. Forward-fix only: pin a known-good digest in `deployment.yaml` (ideally add a tag so Renovate tracks it) + **(BREAK-GLASS)** delete the pod to force re-pull. Lowest priority.
7. **VERIFY on real signal, NOT Kustomization Ready** (intel ks are `wait:false` → Ready=True the instant they APPLY): confirm HR Ready=True, `gpu.intel.com/i915` node-allocatable >0, GPU labels restored, plugin pods Running, and Plex no longer Unschedulable. For nvidia, confirm GPU detection/accounting before declaring the detection-summary pipeline healthy.
8. **BREAK-GLASS** (privileged, escalate to human): (a) plugin pod stuck Terminating/Pending → `kubectl delete pod -n kube-system <pod>`; (b) Plex stuck Pending after i915 returns → `kubectl delete pod -n media <plex>`; (c) HR WEDGED pending-upgrade/retries-exhausted → `flux suspend hr <x> && flux resume hr <x>` or delete the pending Helm storage secret (the single most likely 2am blocker); (d) immutable selector apply failure on the downgrade → `kubectl delete daemonset/deployment <x>`; (e) CRD downgrade blocked by storedVersions → `kubectl patch` the CRD storedVersions; (f) intel CR adoption conflict → re-annotate `meta.helm.sh/release-name`; (g) GPU off the PCI bus → Proxmox HOST reboot, out-of-band.

Caveats: NFD + intel operator install CRDs with `CreateReplace`, and a downgrade IS a Helm upgrade-to-lower-revision — so it actively REPLACES the CRD with the OLDER schema (the storedVersions trap above), gated on whether the bump crossed an apiVersion (rare in 0.18.x/0.36.x minors, real on a major). `git revert <sha>` and a surgical version-only re-pin are NOT equivalent when the PR also changed `values:`/CRs. Intel downgrade order is the REVERSE of dependsOn (gpu plugin down first, then operator) but reverting "both together" lands them in forward order — usually converges. No data-loss risk anywhere; the only un-rollback-able states are the GPU off the bus and `:latest` generic-plugin.

7. Cleanup: once all node work is done + `ceph -s` shows full mon quorum + all OSDs up + active+clean → `ceph osd unset noout` **(BREAK-GLASS** pod exec**)**. A lingering `noout` HEALTH_WARN is expected until cleared.
8. Confirm `git diff` against cluster-template.yaml is empty. Workers may show `uptodate=false` after a live network-config change — cosmetic reboot-pending flag, clears on next reboot.

**Rollback** — *Adversarial verdict:* **converges = conditional (NOT a git→Flux path); data-loss = bimodal.** A `git revert` of cluster-template.yaml does NOTHING on its own — Flux does not manage Talos/k8s, so the Reader SA the gate runs on CANNOT roll this back; the ENTIRE rollback is BREAK-GLASS (Omni admin + OMNICONFIG). **TALOS** patch/minor: in-place A/B downgrade preserves `/var` → NO data loss PROVIDED no node wedges (the re-image fallback for the /boot-too-small nvidia trap on m01/m03/w01 WIPES that node's openebs-hostpath data + erases its Omni identity unless the SMBIOS UUID is pinned first). **KUBERNETES** minor: there is NO safe rollback at all — a minor downgrade is unsupported (apiserver won't start below the stored etcd schema; Talos's controller refuses; note Talos does NOT use kubeadm) → forward-fix-only. The Omni etcd-backup DR lever (`features.backupConfiguration.interval:24h`, template L15-17) is itself lossy (drops state since the last ≤24h snapshot) and cannot restore k8s below the snapshot's storage version.

Corrected procedure:
1. **(GATE 0 — escalate; the scheduled gate STOPS here)** The Reader SA can only DETECT+PAGE. Confirm break-glass is even possible BEFORE touching anything: (a) Sidero-hosted Omni reachable, (b) SideroLink tunnels up / nodes not `connected:false`, (c) you can complete the interactive Omni admin OIDC login (manofoz@gmail.com) from a browser-capable, home-eligible session (Omni OIDC is "only renewable from home" per memory `kubectl-omni-oidc-expiry`). If any is false, you cannot roll back → fall to GATE 5 or wait for home access.
2. **(TRIAGE)** Classify: TALOS-version vs KUBERNETES-version. If the bad commit bumped BOTH, do NOT revert both at once — a k8s-minor decrease is unsupported and will hang the roll. Split.
3. **(TALOS regression — BREAK-GLASS, Omni admin)** Re-pin ONLY `talos.version` down in cluster-template.yaml (commit+push for provenance; inert until sync). `task omni:validate` (schema only — will NOT catch a bad version skew), `omnictl cluster template diff -f ...` to eyeball, then `task omni:sync`. Omni rolls control-planes-first, etcd-quorum + Ceph-PDB gated. Trailing "context deadline exceeded" = normal.
4. **(K8S regression — NO ROLLBACK)** Do NOT sync a `kubernetes.version` decrease: `validate` ACCEPTS it, the sync proceeds, then nodes silently fail to converge (a hung roll after a k8s-down revert is the unsupported-downgrade signature, not a transient timeout). Forward-fix-only — patch forward to a fixed k8s patch, or restore from etcd backup (GATE 5). Never skip minors either direction. Keep k8s 1.36 HELD until Cilium lists support.
5. **(MONITOR — read-only OK)** `omnictl get clustermachinestatus`, `omnictl cluster template status -f ...`, `kubectl get nodes -o wide`, keep `ceph_health_status`==0 between CP reboots. **DANGER WINDOW:** while any CP is wedged/re-imaging you sit at 2/3 etcd quorum with ZERO fault tolerance — serialize CP work; a second CP hiccup loses quorum → apiserver down → the Reader SA goes blind.
6. **(WEDGED NODE — BREAK-GLASS re-image)** If a node reboot-loops (`omnictl machine-logs <id> --log-format dmesg` shows `/boot/B/initramfs.xz: no space left on device`, the small-/boot nvidia trap on m01/m03/w01), the template can't recover it → re-image per `talos-version-upgrade.md`. VM FIRST: `qm set <vmid> --smbios1 uuid=<original-machine-id>` (gotcha #3) and disconnect the .30 VPN NIC (gotcha #5). This DESTROYS that node's openebs-hostpath data. If falling back to direct talosctl, re-mint the cert first: `omnictl talosconfig <file> -c haynes-ops` (it has expired before).
7. **(POST-ROLL FLUX/OPERATOR CLEANUP — BREAK-GLASS writes)** Template-converged ≠ cluster-healthy. Expect + remediate the reboot-roll fallout per memory `talos-reimage-aftermath`: EMQX operator stale-annotation crash-loop, orphaned openebs/VolSync cache PVCs, VolSync stale restic locks, Ceph rule flap. Needs `kubectl delete/annotate` / `restic unlock` / pod restarts.
8. **(CEPH — BREAK-GLASS)** Only after all nodes back AND every OSD/mon rejoined active+clean: `kubectl -n rook-ceph exec deploy/rook-ceph-tools -- ceph osd unset noout`. If a daemon did NOT rejoin, do NOT clear noout — restart the OSD/mon pod first, recover, then unset.
9. **(GATE 5 — LAST RESORT DR, BREAK-GLASS + LOSSY)** If the control plane is bricked or etcd lost quorum, restore from the Omni etcd backup (24h interval). Rebuilds the CP from the last ≤24h snapshot — loses all state since, and CANNOT restore k8s below the snapshot's storage version. Choose the target Talos version in the reverted template before restoring.

Caveats: NOT git→Flux — rollback needs `task omni:sync` (Omni admin write, fully outside the Reader SA). k8s minor downgrades are unsupported (forward-fix-only); you cannot skip minors either direction. Talos in-place up/downgrade preserves `/var`, BUT the re-image fallback WIPES openebs-hostpath + erases Omni identity unless the SMBIOS UUID is pinned. The Cilium/k8s gate is one-directional (never bump k8s past installed Cilium's support — the structural reason 1.36 is HELD). Live network-config edits (e.g. lowercasing a MAC) apply via sync with NO reboot; version bumps reboot every node one at a time and transiently degrade Ceph + reschedule pods.

## flux

The GitOps engine itself — the `flux-manifests` OCI bundle pinning all six gotk controllers (source/kustomize/helm/notification/image-automation/image-reflector) in ns `flux-system`, applied by a SELF-MANAGING `flux` Kustomization. Blast radius: a bad bundle stalls all reconciliation; a regression in source- or kustomize-controller makes the GitOps self-revert IMPOSSIBLE.

- **Renovate match:** datasource `docker`, packageName `ghcr.io/fluxcd/flux-manifests` — the OCIRepository tag at `kubernetes/main/flux/config/flux.yaml` L11 (`ref.tag v2.8.8`). Renovate's NATIVE flux manager detects it; grouped into the `Flux` group by `.renovate/groups.json5` (`matchPackageNames '/fluxcd/'`, `separateMinorPatch:true`) → PR title "Update Flux group". NO `.renovate/autoMerge.json5` rule matches it → never auto-merges → this IS the Tier-4 manual set. The `/fluxcd/` regex is broad but in THIS repo ONLY `flux-manifests` matches (no flux-operator/FluxInstance — do NOT chase `fluxcd/flux-operator`). The bootstrap base `flux2/manifests/install?ref=v2.6.4` (`bootstrap/flux/kustomization.yaml`) is the install floor, EXCLUDED from Renovate via `ignorePaths '**/bootstrap/**'` — it drifts (v2.6.4 floor vs v2.8.8 live) and must be bumped by hand if you care about the repopulation floor. No Flux entry in `holds.json5`.
- **Repo paths:** `kubernetes/main/flux/config/{flux.yaml,cluster.yaml,kustomization.yaml}`, `kubernetes/main/bootstrap/flux/kustomization.yaml`, `kubernetes/main/apps/flux-system/addons/{ks.yaml,app/kustomization.yaml,app/monitoring/podmonitor.yaml,app/monitoring/prometheusrule.yaml,app/webhooks/github/receiver.yaml,app/notifications/github/notification.yaml}`, `.renovate/groups.json5`.
- **Before merging — read:** `gh release view <tag> --repo fluxcd/flux2` (umbrella notes list bundled controller versions + Breaking/Upgrade) — the canonical guide; `https://fluxcd.io/flux/installation/upgrade/` (version-skew) + `https://fluxcd.io/flux/components/` (GA API versions); `https://fluxcd.io/flux/migration/` (API deprecations — the notification Provider/Alert v1beta3→v1 GA migration is the one most likely to land on us); and for a controller called out in the group PR body, that controller's repo releases (flag renames/removals, CRD schema changes).

**Verify (ordered):**
1. BEFORE merge: confirm the flux-local PR gate is green ("Flux Local - Success") — it diffs main+edge and surfaces CRD/schema/field breakage; read the group PR body for which controllers bumped + skim each release's Breaking/CRD-changes.
2. After merge, let Flux apply: the github-receiver fires on push and re-pulls the haynes-ops GitRepository → `cluster` Kustomization → flux.yaml → flux-manifests OCIRepository re-pulls the new tag; without the webhook it lands within the OCIRepository/Kustomization 10m interval. (Do NOT `flux reconcile --with-source`.)
3. `flux get ocirepository flux-manifests -n flux-system` → new tag pulled; `flux get kustomization flux -n flux-system` → Ready=True at the new revision.
4. `flux check` → all 6 controllers "deployment ready" with new image versions + all CRDs OK.
5. `kubectl get pods -n flux-system` → all controllers Running on the new tags, no CrashLoop/restart churn.
6. PromQL: `count(up{namespace="flux-system"})==6` + reconcile error-rate ~0.
7. `flux get kustomizations -A | grep -ivE True` + `flux get helmrelease -A | grep -ivE True` → both empty.
8. Prove forward progress: confirm the next ordinary git change reconciles (or push a no-op annotation) so the upgraded controllers are shown to apply state end-to-end.
9. Optional (repopulation floor): consider manually bumping the renovate-ignored bootstrap ref toward the new version.

**Rollback** — *Adversarial verdict:* **converges = conditional (on WHICH controller regressed); data-loss = LOW.** Flux is stateless — controllers are Deployments (no StatefulSets/PVCs in flux-system), and all durable state (Kustomizations/HelmReleases/sources/Receiver) lives as CRs in etcd that survive a controller revert. The ONE non-trivial risk is a CRD storage-version downgrade: flux-manifests ships CRDs AND controllers as ONE bundle, so reverting the tag re-applies the OLD CRDs; if a newer bundle moved a CRD's storage version and migrated stored CRs, the apiserver may reject removing a version still in `status.storedVersions` → recoverable, but it can hard-wedge the git-only revert. Convergence: ONLY source-controller (pulls both the GitRepository AND the flux-manifests OCI) and kustomize-controller (applies both the `cluster` and self-managing `flux` Kustomizations) are in the revert critical path — and the most-likely regression (a flag rename in the shared `--concurrent/--kube-api-qps` patch) hits source+kustomize+helm SIMULTANEOUSLY, i.e. exactly the controllers that defeat a git-only rollback.

Corrected procedure:
1. **TRIAGE FIRST (read-only, Reader-SA-safe):** `kubectl get pods -n flux-system` + `kubectl logs` (or Grafana) to identify WHICH controllers are crashlooping. The path branches here.
2. **PATH A — regression confined to helm / notification / image-* controllers (NOT critical path):** GIT-FIRST. git revert the flux.yaml tag bump (or re-pin `spec.ref.tag` to last-good v2.8.8), commit, push. A healthy source+kustomize controller self-applies it. Do NOT `flux reconcile --with-source` **(BREAK-GLASS)**. Expect up to ~30-60m if notification-controller is down (gating intervals are 30m GitRepository pull + 30m `cluster` Kustomization, not the 10m the naive read implies). This path CONVERGES from git alone.
3. **PATH B — source- OR kustomize-controller crashlooping (most-likely case):** git-only revert CANNOT converge. **(BREAK-GLASS** — admin kubeconfig, NOT the Reader SA**)**, in THIS order: (1) `flux suspend kustomization flux` to stop the self-managing loop from re-applying the bad bundle; (2) `kubectl rollout undo deploy/source-controller` and/or `deploy/kustomize-controller -n flux-system` to restore the prior working ReplicaSet (smaller delta than re-bootstrap); (3) push the git revert (OCI tag → good) so the now-healthy controllers fetch the good source — the un-suspended `cluster` Kustomization rewrites the OCIRepository tag on its interval/webhook; (4) once `flux get ocirepository flux-manifests` shows the good tag pulled, `flux resume kustomization flux` so it applies the GOOD bundle. WITHOUT the suspend, the `flux` Kustomization (10m) peels your rollout-undo back off → undo/re-apply/re-crash loop.
4. **CRD-BOUNDARY CHECK before any MAJOR revert:** the bundle ships CRDs+controllers together — you cannot revert one without the other. If `status.storedVersions` on a flux CRD contains a version the OLD CRD no longer serves, a clean git revert is REJECTED on the CRD apply. **(BREAK-GLASS)** remediation: keep the newer CRD in place and roll back only the controller image via rollout-undo, or run a kubectl storage migration / re-apply the newer CRD. Watch notification Provider/Alert v1beta3 specifically.
5. **(BREAK-GLASS)** if the bad apply PRUNED CRDs/RBAC (`prune:true`): re-apply the last-good bundle — `kubectl apply -k github.com/fluxcd/flux2/manifests/install?ref=<last-good-tag>` or apply the prior flux-manifests OCI. Symmetric risk: the REVERT apply itself prunes anything the newer bundle ADDED — verify no CRs of newly-added kinds exist before letting prune run.
6. **VERIFY using CLI reads ONLY** (Reader-SA-safe), NOT Prometheus (the PodMonitor `app` label may have gone dark across a major): `flux get ocirepository flux-manifests -n flux-system` (good tag pulled), `flux get kustomization flux -n flux-system` (Ready=True at the revert revision), `kubectl get pods -n flux-system` (all Running/Ready), `flux check` (versions restored). Only after this trust the gotk_*/up metrics again.
7. **PREVENTION (codify):** keep any flux-manifests tag bump in a commit that touches ONLY flux.yaml L11, never co-mingled with flag-patch (L39-86) edits or CR apiVersion bumps, so a `git revert` is surgical.

Caveats: no data loss in Flux itself; CRs persist across a controller revert — the only risk is API/CRD schema mismatch. A crashlooping source/kustomize-controller makes the GitOps self-revert impossible → break-glass (suspend + rollout-undo, NOT git alone) is mandatory. `task flux:bootstrap` re-installs the v2.6.4 FLOOR (a LARGER jump than the regression and possibly incompatible with CRs already migrated to a newer schema) — prefer `kubectl rollout undo` (minimal delta) and reserve re-bootstrap for when controller Deployments/RBAC themselves were pruned/destroyed. The flux.yaml flag-append patches are version-coupled — a `git revert` of a Renovate-style bump touches only the OCI tag (L11), so a hand-edited tag-only rollback that leaves NEW flag edits in place re-triggers the crashloop. The bootstrap ref (v2.6.4) is the renovate-ignored repopulation floor, untouched by a rollback. flux-system carries no PVCs → no VolSync data risk (see `.agents/rules/flux-pvc-prune-safety.md` for the general prune-on-rename caution).
